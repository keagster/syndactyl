Critical Issue: The key exchange (X25519) is the weakest link. If an attacker:
  1. Records encrypted traffic today
  2. Waits for powerful quantum computer (10-20 years)
  3. Uses Shor's algorithm to break X25519
  4. Derives session keys from the handshake
  5. Decrypts all recorded traffic
This is called a "Harvest Now, Decrypt Later" attack.

3. How Quantum Computers Break X25519

Technical Explanation: X25519 relies on the Elliptic Curve Discrete Logarithm Problem (ECDLP), which Shor's Algorithm can solve efficiently on a quantum computer.

Why Classical Computers Can't Do This: Classical computer solving ECDLP (X25519): Time complexity: O(2^128) operations Time required: ~10^25 years (longer than age of universe) Quantum computer with Shor's Algorithm: Time complexity: O((log N)^3) operations Time required: Minutes to hours (estimated) This is an EXPONENTIAL speedup that breaks the security.

6. Implementation Example - Hybrid Mode

Rust Implementation with pqcrypto Crate: // Add to Cargo.toml [dependencies] pqcrypto-kyber = "0.8" // ML-KEM (Kyber) pqcrypto-dilithium = "0.5" // ML-DSA (Dilithium) // Hybrid Key Exchange use pqcrypto_kyber::kyber1024; use libp2p::identity::Keypair as ClassicalKeypair; pub struct HybridKeypair { classical: ClassicalKeypair, // X25519 pq: kyber1024::Keypair, // ML-KEM (Kyber) } impl HybridKeypair { pub fn generate() -> Self { Self { classical: ClassicalKeypair::generate_ed25519(), pq: kyber1024::keypair(), } } pub fn hybrid_key_exchange( &self, peer_classical: &PublicKey, peer_pq: &kyber1024::PublicKey ) -> Vec<u8> { // Derive classical shared secret (X25519) let classical_secret = x25519_key_exchange( &self.classical, peer_classical ); // Derive PQ shared secret (ML-KEM) let (pq_secret, _ciphertext) = kyber1024::encapsulate(peer_pq); // Combine both secrets using KDF let combined = kdf_combine(&classical_secret, &pq_secret); // Security: Protected if EITHER is secure // Classical broken + PQ secure = Still secure // Classical secure + PQ broken = Still secure combined } } // Hybrid Signatures use pqcrypto_dilithium::dilithium5; pub struct HybridSignature { classical: ed25519::Signature, // Ed25519 pq: dilithium5::SignedMessage, // ML-DSA } impl HybridSignature { pub fn sign(message: &[u8], keypair: &HybridKeypair) -> Self { Self { classical: keypair.classical.sign(message), pq: dilithium5::sign(message, &keypair.pq), } } pub fn verify(&self, message: &[u8]) -> bool { // Both signatures must be valid self.classical.verify(message) && dilithium5::verify(&self.pq, message).is_ok() } }
Trade-offs of Post-Quantum Cryptography:
  • Larger keys: ML-KEM public keys: ~1,568 bytes (vs X25519: 32 bytes)
  • Larger signatures: ML-DSA signatures: ~4,595 bytes (vs Ed25519: 64 bytes)
  • Slower: 2-10x slower than classical algorithms
  • More bandwidth: Increased network traffic due to larger messages
  • Less mature: Recently standardized, less battle-tested

‚ö†Ô∏è Harvest Now, Decrypt Later Attack:

Even though quantum computers don't exist yet, adversaries with long-term strategic interest (nation-states, intelligence agencies) may be recording encrypted traffic TODAY with the intent to decrypt it in 10-20 years when quantum computers become available.

Who should worry:

  • Healthcare organizations (HIPAA requires protecting PHI for decades)

Who might not need to worry (yet):

  • Information that loses value within 10 years
  • Non-sensitive data

Summary & Action Items

Current Status: NOT Post-Quantum Secure

  • X25519 key exchange: Vulnerable to Shor's Algorithm
  • Ed25519 signatures: Vulnerable to Shor's Algorithm
  • ChaCha20/Poly1305: Weakened but acceptable (upgrade to 512-bit keys)
  • SHA-256: Weakened but acceptable (upgrade to SHA-512)

Recommended Actions (In Order):

  1. Document the limitation - Add to README that this is not PQ-secure
  2. Assess your threat model - Is "harvest now, decrypt later" a real risk for your use case?
  3. Quick wins - Upgrade to SHA-512, document recommended OS-level encryption
  4. Watch libp2p - Monitor for PQ support in future releases
  5. Plan migration - Design how you'd upgrade when libp2p adds PQ support
  6. 2025-2026 - Implement hybrid mode when available
  7. 2030+ - Full PQ migration

Bottom Line for Most Users:

Unless you're syncing classified documents, trade secrets worth millions, or long-term sensitive data (medical records, etc.), the current encryption is adequate for the next 10 years. The quantum threat is real but not immediate.

However, it's good to plan ahead and be ready to migrate when libp2p adds post-quantum support.
